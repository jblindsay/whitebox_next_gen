# Architectural Complexity Assessment and Burn-Down Plan (2026-05-30)

## Scope

This document compares architectural complexity between:

- Legacy: `whitebox_workflows`
- Next Gen: `whitebox_next_gen`

It provides:

- A complexity rating for each system
- A split between essential complexity and accidental complexity
- Evidence used for scoring
- A prioritized accidental-complexity burn-down checklist with effort estimates

## Executive Summary

- `whitebox_workflows` remains simpler architecturally because it is largely monolithic.
- `whitebox_next_gen` is substantially more complex architecturally, primarily due to intentional modularization and multi-binding/runtime concerns.
- Next Gen complexity is justified for long-term scalability, but some accidental complexity can be reduced without removing capability.

## Ratings (1-10, where 10 is highest complexity)

### Overall

- Legacy `whitebox_workflows`: **6.4 / 10**
- Next Gen `whitebox_next_gen`: **8.8 / 10**

### Split Score

- Legacy essential complexity: **7.6 / 10**
- Legacy accidental complexity: **4.8 / 10**
- Next Gen essential complexity: **8.4 / 10**
- Next Gen accidental complexity: **6.2 / 10**

## Definition of Terms

- **Essential complexity**: Irreducible domain complexity from GIS, raster/vector/lidar processing, topology, projections, and tool semantics.
- **Accidental complexity**: Design/integration overhead not required by the domain itself (cross-boundary duplication, metadata drift risks, runtime mode branching, etc.).

## Evidence and Signals Used

### Topology and Packaging

- Legacy is a single crate with one top-level manifest (`whitebox_workflows/Cargo.toml`).
- Next Gen is a workspace with multiple crates (`whitebox_next_gen/Cargo.toml`) including `wbcore`, `wbraster`, `wbvector`, `wblidar`, `wbtopology`, `wbprojection`, `wblicense_core`, `wbtools_oss`, `wbtools_pro_shim`, `wbw_python`, `wbw_r`.

### Runtime/Abstraction Layering

- Legacy centralizes much behavior in a large monolithic surface (`src/lib.rs`) and tool modules.
- Next Gen introduces explicit runtime abstractions (`ToolRuntimeRegistry`, runtime builders/options/capabilities) and separate binding runtimes.

### Integration Surface

- Legacy: mostly single Rust crate + Python extension model.
- Next Gen: multi-binding and frontend ecosystem (Python, R, QGIS plugin), with metadata contracts crossing boundaries.

### Size/Surface Metrics (observed)

- Rust source files:
  - Legacy: 652 (`whitebox_workflows/src/**/*.rs`)
  - Next Gen: 358 (`whitebox_next_gen/crates/**/src/**/*.rs`)
- Rust LOC:
  - Legacy: 239124
  - Next Gen: 445471 (source paths under crates)
- Direct dependency counts:
  - Legacy crate: 44
  - Next Gen summed across crates: 103

## Why Next Gen Scores Higher

1. Multi-crate orchestration costs:
   - More interfaces, contracts, and ownership boundaries.
2. Runtime and licensing mode branching:
   - Open/Pro/Enterprise tiering and include/expose decisions increase state-space.
3. Cross-language contract management:
   - Tool metadata must stay consistent across Python, R, and QGIS consumption paths.
4. Rich schema/manifest/doc enrichment:
   - Adds quality and flexibility, but also policy logic and reconciliation paths.

## Accidental Complexity Burn-Down Plan

Target: reduce accidental complexity from ~6.2 to ~5.0 without reducing feature scope.

### Priority 1: Canonical Metadata Pipeline

Goal:

- Create a single canonical metadata artifact and policy source generated by core/runtime.
- Bindings/frontends consume this artifact rather than re-deriving order/required/description logic independently.

Current pain points addressed:

- Parameter ordering drift
- Description richness drift
- Enum/required flag inconsistencies

Deliverables:

- Core canonical metadata producer in `wbcore` or tightly-scoped metadata crate.
- Binding/frontend adapters consume canonical output with minimal local transformation.

Estimated effort:

- **5-8 days**

### Priority 2: Shared Binding Adapter Layer

Goal:

- Move duplicated enrichment logic from `wbw_python` and `wbw_r` into one shared Rust module/crate.

Current pain points addressed:

- Duplicate policy maintenance in two bindings
- Regression risk when one binding is updated and the other lags

Deliverables:

- New shared adapter API for:
  - manifest enrichment
  - ordering application
  - schema/description/required merges
- Python and R call the same implementation.

Estimated effort:

- **3-5 days**

### Priority 3: Metadata Contract Golden Tests

Goal:

- Add contract tests that validate exported metadata behavior at runtime boundaries.

Current pain points addressed:

- Silent regressions in parameter descriptions/order/options/required flags

Deliverables:

- Golden or snapshot tests for representative tool families.
- Run for OSS mode and Pro-enabled mode.
- Include checks for:
  - param order
  - non-empty, domain-specific descriptions
  - enum option integrity
  - required flag correctness

Estimated effort:

- **2-4 days**

### Priority 4: Complexity Budget for New Features

Goal:

- Introduce lightweight architecture guardrails in PR/review criteria.

Current pain points addressed:

- Incremental branching and adapter growth without explicit trade-off analysis

Deliverables:

- PR checklist fields:
  - new runtime modes added?
  - new cross-crate coupling introduced?
  - metadata transformations duplicated?
- Reject/reshape changes that increase accidental complexity without clear user value.

Estimated effort:

- **0.5-1 day**

## Proposed Milestones

### Milestone A (Week 1)

- Decide canonical metadata ownership and wire format.
- Build producer and consume in one binding path.

### Milestone B (Week 2)

- Adopt shared adapter in both Python and R bindings.
- Remove duplicated enrichment code.

### Milestone C (Week 3)

- Land metadata contract tests (OSS + Pro).
- Enforce complexity checklist in architecture review workflow.

## Success Criteria

- No duplicated policy logic for metadata enrichment across Python/R.
- Metadata correctness checks fail fast on regressions.
- New tool metadata behavior changes are made once in canonical producer.
- Reduced bug class frequency for:
  - parameter ordering regressions
  - generic/empty parameter descriptions
  - enum population mismatches

## Notes

- This assessment intentionally treats modularization costs in Next Gen as mostly intentional and valuable.
- The burn-down plan focuses only on reducing avoidable complexity while preserving the architectural direction and capability scope.
