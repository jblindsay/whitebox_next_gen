# wbphotogrammetry

Photogrammetry primitives for drone SfM (Structure-from-Motion) workflows.

This is a private crate (Option C) used exclusively by `wbtools_pro`. The API may
be stabilised and selectively open-sourced in a future release after the core
pipeline matures.

---

## Overview

`wbphotogrammetry` implements a complete image-based surface reconstruction pipeline
in pure Rust, designed for small-to-medium drone survey missions (typically tens to
a few hundred frames). The pipeline takes a directory of overlapping aerial images
and produces a georeferenced Digital Surface Model (DSM), an orthomosaic, and a
comprehensive QA report.

The crate has no C or C++ dependencies. All image I/O and numerical computation
uses native Rust libraries.

---

## Pipeline Architecture

The pipeline is organised into six sequential stages. Each stage produces serialisable
output structs that are passed into the next stage, allowing individual stages to be
cached, inspected, or replaced independently.

```
Images on disk
      │
      ▼
┌─────────────┐
│   INGEST    │  Parse EXIF/XMP, compute blur scores, estimate overlap
└──────┬──────┘
       │ ImageFrame[]
       ▼
┌─────────────┐
│  FEATURES   │  Detect keypoints, compute descriptors, match image pairs
└──────┬──────┘
       │ MatchStats + PairCorrespondences
       ▼
┌─────────────┐
│  ALIGNMENT  │  Estimate camera poses, run bundle adjustment
└──────┬──────┘
       │ AlignmentResult (poses + intrinsics + stats)
       ├──────────────────────────┐
       ▼                          ▼
┌──────────────┐          ┌──────────────┐
│    DENSE     │          │    MOSAIC    │
│ (DSM + DTM)  │          │ (Orthomosaic)│
└──────┬───────┘          └──────┬───────┘
       └────────────┬────────────┘
                    ▼
              ┌───────────┐
              │     QA    │  Classify Pass / Review / Fail
              └───────────┘
```

## Modules and Capabilities

### `ingest` — Image Set Ingestion

Scans a directory for supported image formats (JPEG, PNG, TIFF) and builds a
metadata record for each frame.

**Capabilities:**
- EXIF parsing: GPS coordinates (WGS-84 lat/lon/altitude), focal length (mm),
  sensor width (mm), image dimensions, capture timestamp
- XMP orientation priors: standard `GPSImageDirection` / `GPSTrack`, DJI XMP
  extension (`FlightYawDegree`, `GimbalYawDegree`, `FlightPitchDegree`, etc.),
  DJI MakerNote fallback
- RTK GPS flag detection
- Laplacian-variance blur score per frame (higher score = sharper image)
- Overlap estimation from GPS baseline distances and footprint size
- Aggregate ingest QA checks: missing EXIF fraction, blurry-frame fraction,
  overlap percentage estimate, diagnostic warnings

**Key types:** `ImageFrame`, `FrameMetadata`, `GpsCoordinate`, `OrientationPrior`,
`IngestChecks`

---

### `features` — Feature Detection and Matching

Detects corners, computes descriptors, matches image pairs, and validates matches
geometrically. The current implementation supports BRIEF, ORB, an internal
experimental SIFT backend, and `RootSIFT` normalization on top of that SIFT
path, all written directly inside `wbphotogrammetry` without adding new crate
dependencies. The public API still reserves a placeholder hook for a future
SuperPoint backend.

**BRIEF pipeline:**
- FAST corner detection (16-point Bresenham circle, contiguous 9-point brightness
  criterion, non-maximum suppression, 5 px minimum spacing)
- BRIEF descriptor: 4×64-bit binary words from 256 deterministic test pairs,
  generated by a seeded linear congruential generator
- Symmetric Hamming distance matching with Lowe-style ratio test

**ORB pipeline (rotated BRIEF):**
- 4-level Gaussian image pyramid with 1.2× scale factor
- Harris corner score re-ranking to select strongest FAST detections
- Gaussian-sampled rotated BRIEF test pairs; per-keypoint orientation estimated
  from first-order image moments
- Cross-octave match gating (max octave difference = 2)

**SIFT / RootSIFT pipeline:**
- In-house Gaussian scale space and Difference-of-Gaussians extrema detection
- Gradient-histogram orientation assignment
- 128-D floating-point SIFT descriptors with L2 matching
- Optional RootSIFT post-normalization (L1 normalize, elementwise square root,
  final L2 normalize) for improved descriptor distinctiveness under Euclidean matching
- `FeatureMatchingOptions::default()` selects `RootSIFT` with the `balanced`
  profile

**Pair validation pipeline (both variants):**
1. Hamming/L2 filtering with ratio test (requires a valid second-nearest
   candidate; singleton nearest-neighbor cases are rejected)
2. Fundamental matrix RANSAC (8-point normalised DLT + rank-2 SVD enforcement),
   followed by LO-style iterative local re-estimation and inlier refresh
   or robust 2-point translation/scale fitting for near-nadir pairs
3. Spatial distribution filtering (4×4 grid, max 32 matches per cell,
   minimum 2 occupied cells)
4. Orientation and parallax sanity filtering (angle cap, magnitude bounds,
   GPS bearing prior if available)
5. Convergence-point (epipole) filter for cross-flightline pairs

**GPS-aware pair culling:** Frame pairs are prefiltered with a footprint-aware
soft distance weighting around a profile-dependent spacing envelope, with an
outer culling boundary for very distant pairs. This reduces unnecessary
comparison work while preserving near-limit pair candidates at reduced weight.

**Three processing profiles:**

| Profile | Max features | FAST threshold | Match dist | Ratio test | Geom tol |
|---------|-------------|----------------|------------|------------|----------|
| `"fast"` | 450 | 28 | 96 | 0.98 | 18.0 px |
| `"balanced"` | 1 100 | 16 | 78 | 0.96 | 15.0 px |
| `"survey"` | 1 400 | 13 | 60 | 0.92 | 11.5 px |

Small-mission relaxation (≤12 frames): minimum inlier thresholds and spatial
penalty gates are automatically loosened to prevent over-rejection on short
image sequences.

**Key types:** `MatchStats`, `PairCorrespondences`, `AdjacentPairMotion`

---

### `alignment` — Camera Pose Estimation and Bundle Adjustment

Recovers camera positions and orientations from feature correspondences and
refines them via bundle adjustment.

**Intrinsics estimation:**
- From EXIF focal length and sensor width when available
- Identity heuristic (1.2× image dimension as focal length) as fallback

**Network quality scoring:** Weighted combination of inter-frame connectivity
(50%), match density (35%), and feature spatial coverage (15%).

**Initial pose derivation (priority order):**
1. GPS positions when geotag coverage is sufficient
2. Essential matrix decomposition (normalised 8-point algorithm, rank-2 SVD,
   4-candidate Cheirality test) with DLT triangulation for scale
3. Linear trajectory with estimated baseline as last resort

**Bundle adjustment:**
- Levenberg–Marquardt optimisation with Huber loss (2 px threshold)
- Reduced camera translation/rotation normal-equation solves with Schur-style
  point elimination when the tie-point network is strong enough, with guarded
  fallback to conservative local update paths on weak or degenerate systems
- Optional intrinsic refinement: focal lengths, principal point, radial distortion
  (k₁, k₂) and tangential distortion (p₁, p₂) — enabled when the observation
  count is sufficient (≥18 per camera) and network geometry support is strong (≥85%)
- Optional explicit intrinsics policy via `run_camera_alignment_with_options`
  and `AlignmentOptions.intrinsics_refinement`
  (`Auto`, `None`, `CoreOnly`, `CoreAndRadial`, `All`)
- Iterative observation pruning: adaptive residual thresholds derived from RMSE
  quantiles at each pass
- Gaussian pose priors: tighter on small/weakly-supported missions (σ = 2.5 m,
  450 px² scale), looser on well-connected surveys (σ = 6.0 m, 180 px² scale)
- Frozen rotations as a stability fallback for nadir small missions with poor
  geometric support

**Loop closure:** Global trajectory correction using non-adjacent frame pair
constraints to reduce accumulated drift on large or closed-loop surveys.

**Output statistics:** Aligned fraction, reprojection RMSE (median, P95), sparse
point count, track-length statistics, estimated GSD, loop closure correction
magnitude, and covariance-style translation/rotation uncertainty proxies derived
from bundle-adjustment curvature.

**Key types:** `CameraPose`, `AlignmentResult`, `AlignmentStats`, `CameraIntrinsics`,
`CameraModel`, `AlignmentOptions`, `IntrinsicsRefinementPolicy`

---

### `dense` — DSM Reconstruction

Builds a gridded Digital Surface Model from the aligned camera network.

**Grid construction:**
- Regular grid with 2-cell padding around the pose extent
- Requested resolution clamped to [0.02, 20.0] m; falls back to 1.5× estimated
  GSD when no resolution is specified

**Depth hypothesis generation:**
- Trajectory-based: inverse-distance weighted interpolation from camera positions,
  augmented by trajectory-fitted horizontal and vertical slopes
- Stereo-based: coarse patch matching on adjacent post-BA frame pairs, plus
  pose-baseline triangulation for additional hypothesis density

**Surface refinement:**
- Hypothesis curation by confidence score and surface plausibility guards
- Final IDW interpolation with Gaussian smoothing kernel
- Iterative smoothing pass for unusually flat surfaces (range < 0.75 m)

**Outputs:**
- DSM (primary elevation band)
- Per-cell support raster (0–1, observation density)
- Per-cell uncertainty raster (adaptive from support quality and local gradient)
- Optional DTM (morphological low-pass filtering to separate ground from objects)

**Local relief statistics:** Mean and P95 elevation difference within a 3×3
neighbourhood, reported in `DsmStats`.

**Key types:** `DenseResult`, `DsmStats`

---

### `mosaic` — Orthomosaic Generation

Projects source imagery onto the DSM grid to produce a georeferenced orthomosaic.

**Processing steps:**
1. Source image loading and intrinsic/pose extraction
2. Pairwise radiometric refinement: RGB gain ratios estimated from overlapping
   image regions; graph-based iterative balancing
3. DSM cell back-projection: each cell is projected from world space back to
   each source frame using the pinhole camera model; multi-view blending with
   spatial-continuity weighting and per-frame quality factors
4. Seam optimisation: seam mask built from source-image transitions; cost map
   incorporating colour discontinuity and dense-support density; path-based
   seamline finding; soft blending applied across seams

**Auxiliary outputs:**
- Confidence raster (0–1, fused from reprojection accuracy and dense support)
- Support diagnostics raster (source-frame count and dominance ratio per cell)
- Source index raster (1-based source-frame identifier per cell)

**Key types:** `MosaicResult`, `SeamStats`

---

### `qa` — Quality Assessment

Evaluates pipeline outputs against configurable thresholds and classifies the
result as `Pass`, `Review`, or `Fail`.

**Metrics evaluated:**
- Aligned camera fraction
- Reprojection RMSE
- Feature network connectivity
- Mean matches per image pair
- Mean parallax (px)
- Maximum seam colour delta

**Status classification:**
- **Fail:** Any hard failure — no frames, no keypoints, no verified matches,
  < 2 frames, aligned fraction < 50%, or critical feature/parallax deficit
- **Review:** All inputs valid but one or more soft thresholds not met
- **Pass:** All thresholds satisfied

**Three built-in profiles:**

| Threshold | Fast | Balanced | Survey |
|-----------|------|----------|--------|
| Min aligned fraction | 70% | 82% | 90% |
| Max reprojection RMSE | 2.0 px | 1.3 px | 0.95 px |
| Min connectivity | 0.60 | 0.75 | 0.84 |
| Min mean matches/pair | 60 | 90 | 120 |
| Min mean parallax | 1.8 px | 2.8 px | 3.8 px |
| Max seam delta | 0.16 | 0.10 | 0.07 |

Failure codes distinguish hard failures (pipeline-stopping) from soft
degradation indicators. The QA report also carries a list of recommended
corrective actions.

**Key types:** `QaReport`, `QaStatus`, `ProcessingProfile`, `ProfileThresholds`

## Current Limitations

The following limitations reflect the current state of the implementation. They
are documented here to set accurate expectations and to identify the areas most
likely to require further development.

### Feature Matching

- Tuning and benchmarking maturity is still in progress relative to the older
  BRIEF/ORB paths; production users should still validate method/profile
  choices on their mission types.
- `SuperPoint` is still only an API placeholder. Learned inference, weight
  management, and model runtime integration are not implemented.
- SURF is intentionally not on the near-term roadmap because of continuing
  patent risk; it should not be treated as a drop-in alternative to SIFT.
- Cross-scale matching is not supported: ORB matches are gated to adjacent
  pyramid octaves (max difference = 2), which limits robustness when the same
  scene feature appears at very different scales in two frames.

### Camera Alignment and Bundle Adjustment

- Bundle adjustment still uses a simplified Levenberg–Marquardt implementation.
  It now includes iterative structure-only point refresh and coupled
  second-order camera updates, but it still does not expose covariance
  estimates or a full sparse Schur-complement solve. On very large networks
  this can still limit convergence quality and speed.
- Intrinsic refinement now supports selective parameter freezing based on
  observation support/geometry quality, and the refinement policy is now
  user-configurable per run. The default policy remains heuristic rather than
  benchmark-calibrated per mission type.
- Bundle adjustment now evaluates residuals with a fisheye-aware equidistant
  projection path when the fisheye camera model is selected, but intrinsic
  optimization still uses the pinhole-style distortion parameter set
  (`k1/k2/p1/p2`). A dedicated fisheye calibration parameterization is not yet
  implemented.
- Camera-motion priors (smooth acceleration, constant-speed models suited to
  drone flight paths) are not implemented. The current pose priors are
  Gaussian position constraints only.
- The loop closure optimisation uses a basic graph solver with non-adjacent
  pair constraints and robust correction-magnitude outlier rejection. It is
  not a full pose-graph SLAM formulation.
- The essential matrix recovery is based on the 8-point normalised DLT
  algorithm alone. It now rejects clearly weak/degenerate seeds and falls back
  to the simpler match-driven trajectory seed, but there is still no iterative
  triangulation refinement or richer multi-model degeneracy handling.

### Dense Surface Reconstruction

- The crate does not implement multi-view stereo (MVS). Stereo depth estimation
  relies on coarse patch matching between adjacent post-BA frame pairs and
  pose-baseline triangulation. This produces surfaces with lower point density
  and surface completeness than a full MVS approach (patch-based or depth-map
  fusion). In particular:
  - No depth-map voting or occlusion handling
  - No cross-baseline epipolar line search
  - No semi-global matching (SGM/SGM++), Census transform, or learned cost volumes  
- DTM separation is a morphological low-pass filtering operation. It is not a
  classification-based or progressive triangulation approach and will not
  reliably separate buildings, dense vegetation, or tall structures from the
  ground surface on complex terrain.
- Vertical accuracy is estimated heuristically from GSD and support density.
  There is no mechanism for absolute accuracy validation against GCPs or an
  independent reference surface.
- Water, transparent, and low-texture surfaces (sand dunes, snow, uniform crops)
  will produce unreliable depth hypotheses because stereo patch matching
  requires photometric contrast.
- The surface plausibility guards and flat-surface reinforcement thresholds
  (e.g., range < 0.75 m treated as "too flat") are fixed constants that have
  not been validated across diverse terrain types.

### Orthomosaic

- Radiometric correction is limited to per-image RGB gain adjustment derived
  from overlap regions. There is no per-channel white balancing, no
  bidirectional reflectance correction (BRDF), and no absolute radiometric
  calibration.
- Seamline optimisation uses a cost-map path-finding approach without
  gradient-domain or Poisson blending. Visible colour boundaries may remain
  near seams in scenes with strong radiometric difference between adjacent
  images.
- Only the standard RGB bands are supported. Multispectral, near-infrared,
  thermal, and hyperspectral bands are not handled.
- Edge pixelation can occur near image boundaries where the back-projection
  lands close to the sensor edge and bilinear sampling wraps into the border.

### General and Architectural

- The pipeline is single-process and in-memory. There is no tiling or streaming
  support for surveys larger than available RAM; very large image sets (≥ 500
  frames at full resolution) may encounter memory pressure.
- All processing profiles (`"fast"`, `"balanced"`, `"survey"`) use fixed
  threshold constants. There is no adaptive tuning from previous project runs
  or learned calibration.
- The `PhotogrammetryError::NotImplemented` variant is present as a development
  marker; any code path that returns it is explicitly unfinished.
- The GPS debug filter (`WBPHOTOGRAMMETRY_DEBUG_GPS_FILTER` environment
  variable) is a development diagnostic. It is not an end-user feature and
  produces verbose `eprintln!` output to stderr.

## Running the Examples

```bash
# Profile the full pipeline on a flight directory
cargo run --example profile_pipeline --release -- --profile balanced --feature-method rootsift

# Inspect feature matching on a specific image pair
cargo run --example feature_pair_probe --release -- --images-dir <dir> --left <image_a> --right <image_b> --method rootsift

# Compare QA reports across profiles (requires Python 3)
python3 examples/compare_profile_reports.py <report_fast.json> <report_survey.json>

# Batch benchmark across multiple input sets
bash examples/run_profile_benchmark.sh
```
